Composite workflows are the Rust-side orchestration wrappers generated by `composite_workflow!`.
They coordinate one or more workflow invocations and local orchestration logic inside a Tokio task.
Status note: normal/composite workflow separation remains useful target vocabulary; this page documents the legacy
macro-generated implementation shape, not a frozen future API.

## Current Behavior (Macro-Generated)

1. `composite_workflow!` normalizes name/capture syntax and emits an internal async function via
   `define_composite_workflow!`.
2. Captured values use directional semantics:
    - `move in`: pass into scoped context
    - `move out`: read back from scoped context on completion
    - `move` (both): pass in and read back out
3. Fallible composite mode is auto-selected when fallible `workflow!(E|OE|IE|IOE ...)` invocations are detected.
4. Execution is spawned on workflow Tokio runtime through `CompositeWorkflowRuntime`.
5. Scoped composite context is created per composite run and cleared by return handlers after completion.

## Current Return Handling Surface

- `handle_composite_workflow_return_now`:
    - used when caller expects completion now and handles an already-finished join handle.
- `handle_composite_workflow_return_later`:
    - spawns callback handling later on workflow runtime and clears context after callback.
